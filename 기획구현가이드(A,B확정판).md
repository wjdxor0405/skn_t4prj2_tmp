## 0. 데이터

- **데이터셋**: IBM Telco Customer Churn 원본(blastchar), 7,043건, 21개 컬럼
- **확장판(Churn Reason 포함)**: 메인 분석에는 미사용. 시간 남으면 사후 검증(삼각검증)용으로만 활용 (보너스, 우선순위 낮음)
- **실데이터 확인된 핵심 수치**: 이탈 1,869건(26.5%), TotalCharges 결측 11건(전부 tenure=0, 전부 비이탈), No internet service 1,526명(21.7%), No phone service 682명(9.7%)

**⚠️ 모델의 본질적 한계 (학습 결과서에 반드시 명시할 것)**
- 이 데이터는 절대 날짜(가입일/이탈일)가 없는 **스냅샷 데이터**(tenure=경과월, Churn=현재 이탈여부만 존재). tenure별 집계는 한 개체가 시간에 따라 전이되는 시계열이 아니라, 서로 다른 개체들의 횡단면 스냅샷임
- 모델의 본질은 "몇 개월 후 이탈할지" 예측이 아니라 **"현재 속성 기준 이탈 위험도 점수" 산출**. 분석 A/B는 그 점수가 왜 높은지를 생애주기(A)·속성(B) 두 축으로 설명하는 역할 — 점수를 만드는 게 아니라 점수의 근거를 제공
- "언제 이탈하는가"는 분석A(위험구간)+분석B(속성위험도) 결합으로 **간접 추정**만 가능. 리드타임도 절대 시간이 아니라 tenure 구간 단위로 정의됨

## 1. 전체 파이프라인 순서

```
원본 로드 → 자료형 정리 → 결측치 처리 → 중복정보 통합(No internet/phone service)
→ Train/Test 계층화 분할
→ [분석A 반복 사이클, Train만]
    ① 가지치기 회귀나무로 세그먼트 경계 탐지 (tenure 단일변수)
    ② 세그먼트단독 AUC + 순열검정으로 적절성 검증
    ③ AUC 부트스트랩 신뢰구간으로 표본충분성 점검
    └─ ②에서 우연(p값 높음) 또는 ③에서 신뢰구간 넓음(표본부족) → 문제 구간을 인접 구간과 통합 후 ①부터 재실행
    └─ ②③ 모두 통과 → 경계 확정, 다음 단계로
→ segment 컬럼 추가(Train/Test 양쪽) → 인코딩(segment 포함 원-핫, Train 기준 더미컬럼을 Test에 강제 정렬) → 스케일링(트리용 원본 / 선형용 표준화, 두 버전)
→ 분석B: 세그먼트별 속성-이탈률 관계 탐색 → 보조Q: 전체데이터 클러스터링
→ 모델 학습(1·2·3단계) → 평가 → SHAP 해석 → 결과를 df에 재결합
```

**핵심 원칙**: pandas DataFrame 하나(`df`)를 끝까지 유지. customerID 등 식별 컬럼을 위한 별도 테이블을 만들지 않음. 모델 입력 `X`는 필요한 시점에 `df`에서 컬럼만 선택. 예측 후 결과는 다시 `df`에 새 컬럼으로 합침.

**누수 방지 원칙(일관 적용)**: 세그먼트 경계, 더미 컬럼 목록, 스케일러의 fit은 전부 **Train으로만 수행**하고 Test에는 적용만 한다.

## 2. 전처리 세부 사항

| 단계 | 처리 방법 |
|---|---|
| TotalCharges 자료형 | `pd.to_numeric(errors='coerce')`로 숫자 변환 |
| TotalCharges 결측(11건) | 0으로 대체 (tenure=0 신규고객, 아직 청구 전) |
| customerID | 모델 입력 X에서는 제외, df 자체에는 끝까지 보존(결과 매칭·손해추정용) |
| 이진 범주형 (gender, Partner 등) | `.map({'Yes':1,'No':0})` |
| Contract | 원-핫 인코딩 (순서형 대신 — 선형모델 비교군 호환 위해) |
| 기타 다중범주형 (InternetService, PaymentMethod 등) | 원-핫 인코딩 |
| "No internet service" / "No phone service" | "No"로 통합 (이유: 정보손실 방지가 아니라 InternetService_No/PhoneService_No와 100% 중복되어 원-핫 시 다중공선성 유발 방지) |
| **segment (분석A가 만든 세그먼트 라벨)** | **원-핫 인코딩** (Contract와 동일하게 다중범주형 취급 — 6장 예측모델 피처로 쓰이므로 누락 금지) |
| 연속형(tenure, MonthlyCharges, TotalCharges) | 트리모델용: 원본 그대로 / 선형·MLP용: `StandardScaler`(Train fit, Test transform) — 두 버전 병행. 시간 부족시 생략 가능(메인 서사에 지장 없음) |
| Train/Test 분할 | `train_test_split(stratify=Churn, test_size=0.3)` |
| 더미 컬럼 정렬 | df_train 기준 생성된 더미 컬럼을 df_test에 강제로 동일하게 정렬 |

## 3. 분석 A — 세그먼트 경계 탐지 (최종 확정 구조, ①②③ 3단계)

> ⚠️ 이 섹션의 ①②③은 "예측모델의 1·2·3단계"(5번 섹션)와는 다른, **분석 A 내부의 절차 번호**입니다. 혼동 주의.

**① 경계 탐지: 가지치기 회귀나무 (메인, 머신러닝) + 랜덤포레스트 투표 (보조 검증)**
- `sklearn.tree.DecisionTreeRegressor` + `cost_complexity_pruning_path`로 ccp_alpha 후보 산출
- 5-fold 교차검증(예측오차 최소화)으로 최적 ccp_alpha 선택 — 가지치기 강도를 사람이 직감으로 안 정하고 데이터가 직접 결정
- 입력: tenure 단일변수, 출력: tenure별 평균 이탈률(가중치는 각 tenure의 표본수 `sqrt(count)` 권장)
- **⚠️ 필수 구현 규칙**: `cost_complexity_pruning_path`가 반환하는 alpha 후보를 절대 샘플링하지 말고 전체를 다 탐색할 것 — 일부만 샘플링하면(예: 절반만) 탐색 그리드 설정에 따라 결과 K값이 미세하게 흔들리는 그리드 민감성이 실제로 확인됨
- **보조 검증(랜덤포레스트 투표)**: `RandomForestRegressor`(n_estimators=200~300, 같은 입출력)를 학습시켜 모든 개별 트리의 분기점(threshold)을 전부 수집·빈도 집계. CV 성능 상위 alpha 후보들의 경계가 이 투표 결과의 상위 후보와 일치하는지 확인 — 일치하면 신뢰, CV성능은 높은데 투표에서 안 나오는 경계라면 더 보수적인 후보로 한 단계 내려갈 것
  - 단일 결정나무(정밀한 좌표, 그리드에 다소 민감) vs 랜덤포레스트 투표(좌표는 덜 정밀하나 순위가 시드 변경에 안정적)는 정밀도-안정성 트레이드오프 — **랜덤포레스트로 ①을 통째로 대체하지 말 것**, 보조 검증으로만 활용
- 통계적 비교군(PELT+BIC, RBF커널 PELT, 가우시안프로세스회귀)과도 결과를 비교해 핵심 구간이 독립적으로 수렴하는지 확인 — 상호 교차검증 역할 (구체적 경계 수치는 코드 실행 후 확인)

**② 적절성 검증: 세그먼트단독 AUC + 순열검정 (메인 검증, 데이터 재배열 기반)**
- ①이 찾은 세그먼트 라벨만(다른 속성 없이) 입력으로 분류모델(RandomForestClassifier) 학습, 5-fold 교차검증 AUC 측정
- 이탈여부 라벨을 수백 회(200회 이상) 무작위로 섞어 같은 절차 반복 → 실제 AUC가 그 분포보다 압도적으로 높으면 경계가 통계적으로 실재
- 실제 AUC가 무작위 섞기로 나온 AUC 분포보다 압도적으로 높고 p값이 0에 가까우면 경계가 통계적으로 실재 — 이론적 분포(카이제곱 등) 전혀 사용 않는 완전 데이터기반 검증 (구체적 수치는 코드 실행 후 확인)
- **중요**: 이 검증은 "경계 자체가 실재하는가"를 보는 것이며, "예측모델(20개 변수 포함)에 추가했을 때 분류성능이 오르는가"와는 다른 질문(5번 참조). 후자는 노이즈에 묻혀 측정 안 됨 — 혼동 주의

**③ 표본충분성 확인: AUC 측정값의 부트스트랩 신뢰구간**
- ②에서 측정한 세그먼트단독 AUC를 부트스트랩(수백 회)으로 재추정해 신뢰구간 계산
- 신뢰구간이 좁으면(예: ±0.02) 그 표본 크기에서 측정이 안정적. 넓으면(예: ±0.15 이상) 표본부족 신호 — 인접 구간과 통합 후 ②부터 재검증
- 사전 외부 공식(검정력분석 등) 없이, 측정 자체의 안정성을 사후에 직접 확인 — 4번 상세

**①②③의 관계 (혼동 주의, 완전히 별개 절차)**: ①이 머신러닝이라고 ②가 자동 보장되지 않음 — 실제 검증했기 때문에 통과를 확인한 것. ②(세그먼트~이탈여부 관련성)와 ③(AUC 측정 자체의 안정성)도 서로 다른 질문 — 코드 구현 시 반드시 별개 함수로 작성

**기각된 대안 기법들 (시도 후 부적합 확인, 재시도 불필요)**
- K-means/일반 클러스터링: 시간 순서를 모르고 값의 유사성만 봐서 라벨이 뒤섞임
- LSTM/딥러닝 시계열기법: 시계열 샘플 1개·길이 73뿐인 극단적 데이터 빈곤
- 가지치기 없는 결정나무: 페널티 없어 K=10 이상 과도 세분화
- HMM(은닉마르코프모델): "상태 전이" 가정이 횡단면 스냅샷 데이터와 안 맞음
- 평활화 스플라인: 교차검증 최적 매끄러움에서 변화점이 전혀 검출 안 됨 (예측오차 최적화≠변화점 가시화)
- 분류모델 교차검증으로 K값 자체 탐색: 여러 시드 반복 시 신호 사라짐(노이즈)
- 카이제곱 검정(메인 검증으로): 이론적 분포 의존, 기대빈도 부족 시 신뢰도 저하 — 순열검정으로 대체
- Cohen 효과크기 기반 검정력분석(392건): 카이제곱 시절의 유산, 도입 근거 소멸 — AUC 신뢰구간으로 대체

**세그먼트 이름**: 사전에 "신규/중간/장기"로 고정하지 않음. 경계 확정 후 분석 B로 각 구간 성격을 파악한 다음 사후에 이름 부여 (임시로는 "구간1, 구간2..." 사용)

## 4. 표본충분성 기준 (최종: AUC 부트스트랩 신뢰구간으로 직접 확인)

**⚠️ [정정] Cohen 효과크기 기반 검정력 분석(392건)은 폐기 — 더 이상 코드에 쓰지 않음**
- 이 기준은 메인 검증을 카이제곱 검정으로 가정했을 때, 기대빈도 부족으로 인한 신뢰도 저하를 막기 위해 도입한 것
- 메인 검증이 순열검정(이론적 분포 가정 자체가 없는 재배열 검정)으로 확정되면서 도입 근거가 사라짐 — `NormalIndPower` 등 검정력 분석 함수를 코드에 쓰지 말 것

**남는 진짜 문제와 해결책**: 표본이 너무 적으면 분류모델이 패턴을 안정적으로 학습 못해 AUC 측정 자체가 불안정해짐 (사전 외부 공식으로 막을 수 없는, 머신러닝 고유의 문제). 대안:
- 각 세그먼트(또는 K 후보)의 세그먼트단독 AUC를 부트스트랩(수백 회)으로 재추정해 **신뢰구간**을 계산
- 신뢰구간이 좁으면(예: ±0.02 수준) 그 표본 크기에서 측정이 안정적이라고 직접 확인. 넓으면(예: ±0.15 이상) 표본부족 신호 — 인접 구간과 통합 후 재검증
- 절대임계값(신뢰구간 폭 몇 %p 이상이면 불안정)은 자동 탐지하지 않음 — K별 신뢰구간 표를 투명하게 제시하고 사람이 종합 판단

**⚠️ 3번의 ②(적절성검증)·본 섹션(③ 표본충분성)·7번(경계좌표 안정성검증)의 부트스트랩 — 세 가지 별개 목적**
- 3번 ②: 세그먼트단독 AUC의 순열검정(라벨 재배열) — 경계가 이탈여부와 실재 관련있는지 검증
- 본 섹션(3번 ③): AUC 자체의 부트스트랩 신뢰구간 — 그 측정이 표본크기 면에서 안정적인지 확인
- 7번: 경계 좌표 자체가 재추출해도 같은 위치에서 나오는지 (안정성 검증)
- 코드 구현 시 세 가지 부트스트랩/순열검정 함수를 분리해서 작성 (목적이 다름, 절대 하나로 합치지 말 것)

## 5. 예측모델 설계

> ⚠️ 아래 "1·2·3단계"는 3번(분석A)의 ①②③과는 다른, **예측모델 진화서사의 번호**입니다(7번 참조). 혼동 주의.

**입력 피처 (확정)**
- 전체 속성 + **tenure 원본** + **세그먼트 라벨** (둘 다 사용, 양자택일 아님)
  - tenure 원본: 세밀한 정보 보존
  - 세그먼트 라벨: 분석 A 발견의 기여도를 SHAP으로 직접 검증 가능
  - 다중공선성 위험 인지하나 트리 기반 모델은 강건, 1단계 비교군(선형모델)만 영향 받음(치명적이지 않음)

**3단계 비교 구조 (입력 차이를 "세그먼트 라벨 포함 여부" 하나로 통제)**

| 단계 | 알고리즘 | 입력 피처 | 역할 |
|---|---|---|---|
| 1단계 | 로지스틱 회귀 | 전체 속성 + tenure 원본 (스케일링됨) | 전통통계 비교군 — 세그먼트 라벨 절대 미포함 |
| 2단계 | XGBoost | 전체 속성 + tenure 원본 (세그먼트 라벨 미포함) | 현재AI 비교군 |
| 3단계 | XGBoost (2단계와 동일 알고리즘) | 전체 속성 + tenure 원본 + **세그먼트 라벨** | **메인, 우리의 접근** |
| 보조 | 간단한 MLP | 2단계와 동일 입력 (스케일링됨) | 딥러닝 비교용 (부가) |

**왜 1단계에도 세그먼트 라벨이 없는가**: 1단계는 "사람이 사전에 정한 단순 함수형태에 의존"하는 전통통계 대표 비교군. 세그먼트 라벨 자체가 데이터 주도(분석 A)로 찾은 결과물이므로 이를 1단계에 넣으면 정의와 모순

**왜 2·3단계가 같은 알고리즘인가**: 성능 차이가 "접근법" 때문인지 "알고리즘" 때문인지 혼동되지 않게 하기 위함

**⚠️ 분석A의 알고리즘(①회귀나무·②랜덤포레스트)과 여기 XGBoost가 다른 이유**: 3번의 ①②는 "경계가 실재하는지 가볍고 표준적인 도구로 빠르게 검증"하는 목적이고, 여기 XGBoost는 "최종 산출물의 메인 예측 성능"을 위한 선택 — 목적이 다른 별개 모델이라 알고리즘이 다른 게 정상. 3번 ②에서 나온 "세그먼트단독 AUC"는 랜덤포레스트 기준 수치이며, 아래 2·3단계 비교(XGBoost)의 결과와는 별개 모델임을 보고서에 명시할 것

**⚠️ [중요] 2·3단계 분류성능 비교의 정확한 위치 (코드 실행 후 직접 확인할 것)**
- 세그먼트 라벨을 20개 속성이 이미 들어간 모델에 추가했을 때, F1 증분 기여가 작게(또는 거의 없이) 나올 가능성이 있음을 미리 인지하고 진행 — tenure 원본이 이미 비슷한 정보를 담고 있어 증분 기여가 작아질 수 있는 현상(다중공선성과 유사). 단일 시드 결과만으로 판단하지 말고 여러 시드로 재현되는지 반드시 확인
- **만약 증분 기여가 작게 나와도 "세그먼트가 무의미하다"는 뜻이 아님**: 분석 A의 적절성 자체는 3번에서 별도로 검증됨(세그먼트단독 AUC + 순열검정)
- 따라서 2·3단계 분류성능 비교는 **"세그먼트 발견의 증거"가 아니라 예측모델 차원의 보조적 실용성 지표**일 뿐. 성능차이 있으면 실용적 부가가치로 보고, 없어도 분석A의 별도 검증(3번)과 SHAP 해석으로 서사 유지 — 메인 증거는 3번임을 보고서에 명확히 할 것

**보조 분석 Q (클러스터링, 메인 모델과 무관)**
- tenure+속성을 함께 K-means/GMM으로 클러스터링 (전체 데이터, 세그먼트 구분과 무관하게 1회 수행)
- 목적: "어떤 속성 조합이 결합되면 위험한가"라는 인사이트만 추출, 새 세그먼트로 채택 안 함
- 모델 입력에는 쓰지 않음, 발표/해석 보강용

## 6. 평가지표 및 임계값

**보고서에는 전부 계산해서 표로 제시**: Accuracy, Precision, Recall, F1, F2, ROC-AUC, PR-AUC
- Accuracy는 참고용으로만 표시 (전부 비이탈로 찍어도 73.5% 나오는 불균형 데이터)

**⚠️ 용어 구분 — "AUC"가 두 군데서 다른 대상**: 여기(6번)의 ROC-AUC는 **최종 예측모델(20개+ 변수 전체)의 분류 성능**. 3번 ②의 "세그먼트단독 AUC"는 **세그먼트 라벨 하나만의 판별력**으로 완전히 다른 수치(값은 코드 실행 후 확인). 코드 변수명·보고서 표기 모두 `model_auc`/`segment_only_auc` 등으로 구분해서 절대 혼용하지 말 것

**모델 튜닝 목적함수**: F2-score(Recall에 2배 가중치) 우선, Recall 보조 모니터링
- 이유: FN(이탈자 놓침)이 FP보다 비싼 비용이라는 가정 (정확한 비용비율은 미해결, 가정값임을 명시)
- **베타값(2)과 비용비율의 관계**: F-beta의 베타는 비용비율을 정밀하게 환산한 수치가 아니라 근사적 가중치. 베타=2는 Recall을 Precision보다 대략 4배(베타²) 중요하게 취급한다는 의미일 뿐, "5:1" 같은 구체적 비용비율과 수학적으로 정확히 대응하지 않음
- **FN비용은 근사추정 가능**(비이탈고객 평균tenure와 이탈고객 평균tenure 차이 × MonthlyCharges로 계산), **FP비용(리텐션비용)은 데이터에 정보 자체가 없어 추정 불가** — 비용행렬(cost matrix)도 FP비용 부재로 F2보다 우위 없음. F2를 그대로 채택, 근거는 "정보부재를 검증으로 확인"으로 보강 (구체적 FN비용 추정치는 코드 실행 후 확인)

**분류 임계값**: 0.5 고정 대신, Precision-Recall 곡선에서 "Recall이 급격히 꺾이기 직전 지점" 채택

## 7. 3단계 비교 — 실증 책임 (반드시 검증할 것)

1. **(메인 증거) 분석A 적절성 검증**: 가지치기 회귀나무(①)가 찾은 경계의 세그먼트단독 AUC + 순열검정(②) 결과 보고, AUC 부트스트랩 신뢰구간(③)으로 측정 안정성도 함께 제시, 경계 좌표 자체의 재현 안정성도 별도로 확인(재추출 1,000회 권장)
2. **1단계(전통통계)와 비교**: Kaplan-Meier(`lifelines`)·PELT(`ruptures`)와 가지치기 회귀나무 결과 비교 — 핵심 구간이 유사하게 수렴하는지 확인, 상호 교차검증 역할
3. **2단계(현재AI, 단순 피처투입형)와 비교**: 동일 데이터에 2단계(세그먼트 라벨 없음)와 3단계(세그먼트 라벨 포함)를 모두 적용해 분류성능 비교
   - ⚠️ 이 비교는 보조적 실용성 지표일 뿐, "세그먼트 발견의 증거"가 아님(5번 참조) — 메인 증거는 1번
   - 성능 차이 없으면(실제로 그럴 가능성 높음): SHAP으로 세그먼트 라벨 피처의 중요도를 보여주며 해석적 가치 강조, "성능이 아니라 구조를 보여준다"는 서사로 전환
4. **해석 보강**: 분석 B 결과로 "왜 이 경계 지점인가" 설명

## 8. 분석 B 및 보조 Q 구현 메모

**메인 기법 확정 — 분석A와 동일한 2단 구조**

| 단계 | 방법 |
|---|---|
| Ⓐ 패턴 탐지 | 세그먼트별 가지치기 결정나무(`DecisionTreeClassifier`, 얕은 깊이+`min_samples_leaf`) 또는 랜덤포레스트 피처중요도로 주요 위험속성 탐지 |
| Ⓑ 적절성 검증 | 그 속성들만으로 만든 모델의 AUC 측정 + 라벨 순열검정 (분석A의 ②와 동일 절차) |

**확인할 것 (코드 실행 후)**: 세그먼트마다 주요 위험속성·AUC·순열검정 p값을 표로 정리. "같은 속성이라도 생애주기 맥락에 따라 영향력이 다르다"는 분석B 핵심가치가 실제로 드러나는지, 각 세그먼트 AUC가 충분히 높고 순열검정 p값이 낮은지(패턴이 노이즈가 아닌지) 확인.

**클러스터링·교차표 위치**: 클러스터링은 보조Q(전체데이터 1회)로 이미 확정되어 분석B 메인으로 다시 쓰지 않음. 교차표는 결정나무 결과를 보완하는 시각적 보조자료로만 사용 — "왜 중요한지"는 모델이, "구체적 수치"는 교차표가 보여주는 역할 분담.

**순서**: 분석 A(경계 탐지+검증) → 세그먼트 확정 → 분석 B(세그먼트별 Ⓐ→Ⓑ) → 보조 Q. Q는 분석 A/B와 완전히 무관하게 전체 데이터로 1회만 수행
