## 0. 데이터

- **데이터셋**: IBM Telco Customer Churn 원본(blastchar), 7,043건, 21개 컬럼
- **확장판(Churn Reason 포함)**: 메인 분석에는 미사용. 시간 남으면 사후 검증(삼각검증)용으로만 활용 (보너스, 우선순위 낮음)
- **실데이터 확인된 핵심 수치**: 이탈 1,869건(26.5%), TotalCharges 결측 11건(전부 tenure=0, 전부 비이탈), No internet service 1,526명(21.7%), No phone service 682명(9.7%)

**⚠️ 모델의 본질적 한계 (학습 결과서에 반드시 명시할 것)**
- 이 데이터는 절대 날짜(가입일/이탈일)가 없는 **스냅샷 데이터** (tenure=경과월, Churn=현재 이탈여부만 존재)
- 따라서 모델은 엄밀한 미래 예측이 아니라 **"현재 속성으로 현재 이탈여부를 분류"**하는 모델임. 신규고객에게 적용 시 "이 속성을 가진 고객들이 결국 이탈할 가능성"으로 해석되어 사실상 예측 효과를 내지만, 정확한 "이탈까지 남은 시간"은 직접 예측하지 않음
- "언제 이탈하는가"는 분석A(위험구간)+분석B(속성위험도) 결합으로 **간접 추정**만 가능. 리드타임도 절대 시간이 아니라 tenure 구간 단위로 정의됨

## 1. 전체 파이프라인 순서

```
원본 로드 → 자료형 정리 → 결측치 처리 → 중복정보 통합(No internet/phone service)
→ Train/Test 계층화 분할 → 세그먼트 경계 산출(Train만, PELT+BIC) → segment 컬럼 추가
→ 인코딩(Train 기준 더미컬럼을 Test에 강제 정렬) → 스케일링(트리용 원본 / 선형용 표준화, 두 버전)
→ 모델 학습(1·2·3단계) → 평가 → SHAP 해석 → 결과를 df에 재결합
```

**핵심 원칙**: pandas DataFrame 하나(`df`)를 끝까지 유지. customerID 등 식별 컬럼을 위한 별도 테이블을 만들지 않음. 모델 입력 `X`는 필요한 시점에 `df`에서 컬럼만 선택. 예측 후 결과는 다시 `df`에 새 컬럼으로 합침.

**누수 방지 원칙(일관 적용)**: 세그먼트 경계, 더미 컬럼 목록, 스케일러의 fit은 전부 **Train으로만 수행**하고 Test에는 적용만 한다.

## 2. 전처리 세부 사항

| 단계 | 처리 방법 |
|---|---|
| TotalCharges 자료형 | `pd.to_numeric(errors='coerce')`로 숫자 변환 |
| TotalCharges 결측(11건) | 0으로 대체 (tenure=0 신규고객, 아직 청구 전) |
| customerID | 모델 입력 X에서는 제외, df 자체에는 끝까지 보존(결과 매칭·손해추정용) |
| 이진 범주형 (gender, Partner 등) | `.map({'Yes':1,'No':0})` |
| Contract | 원-핫 인코딩 (순서형 대신 — 선형모델 비교군 호환 위해) |
| 기타 다중범주형 (InternetService, PaymentMethod 등) | 원-핫 인코딩 |
| "No internet service" / "No phone service" | "No"로 통합 (이유: 정보손실 방지가 아니라 InternetService_No/PhoneService_No와 100% 중복되어 원-핫 시 다중공선성 유발 방지) |
| 연속형(tenure, MonthlyCharges, TotalCharges) | 트리모델용: 원본 그대로 / 선형·MLP용: `StandardScaler`(Train fit, Test transform) — 두 버전 병행. 시간 부족시 생략 가능(메인 서사에 지장 없음) |
| Train/Test 분할 | `train_test_split(stratify=Churn, test_size=0.3)` |
| 더미 컬럼 정렬 | df_train 기준 생성된 더미 컬럼을 df_test에 강제로 동일하게 정렬 |

## 3. 분석 A — 세그먼트 경계 탐지 (시간축)

**방법론적 위치 (보고서 작성 시 반드시 구분해서 서술)**
- 1단계(PELT+BIC/AIC)는 **순수 통계학** — 훈련 데이터로 일반화를 학습하는 과정이 없고, 주어진 분포에 대한 1회의 수학적 최적화. "AI/머신러닝"이라고 부르면 안 됨
- 2단계(교차검증)는 **머신러닝** — 분류 모델이 실제로 데이터에서 패턴을 학습해 일반화 성능을 평가
- 따라서 "경계 탐지 = 머신러닝"이라는 표현은 부정확. 정확한 설명: "통계가 경계 후보를 제안하고, 머신러닝이 그 후보의 실용성을 검증해서 최종 선택"

**입력**: df_train의 tenure별 이탈률 분포만 (이탈 여부와 무관하게 시간 분포만 봄 → 누수 방지)

**1단계 — 통계적 변화점 탐지**
- 라이브러리: `ruptures` (`rpt.Pelt(model="l2")`)
- 모델선택 기준: BIC 기본 사용 (AIC보다 페널티가 강해 표본 적은 우리 데이터에 더 안전)
- 최소 세그먼트 길이: 이탈 **392건**(4번의 사전 범용 기준) 역산값을 **느슨한 시작값**으로 사용 → 약 **15.3개월**(전체 73개월의 21%). 단, 이 값은 전체 평균 기반 근사치이며 구간별 실제 안정성을 보장하지 않음 (분석 A 실제 데이터에서 0-17개월/44-72개월의 이탈률이 54%/8%로 크게 달라 평균 가정이 깨짐). 정밀 검증은 2단계가 담당
- 탐색 K 범위: 위 최소 길이에서 "전체기간÷최소길이"로 역산한 기술적 상한 약 **K=4~5**를 참고하되, 사전 상한으로 강제하지 않고 **K=2~8 정도로 넓게 탐색** (운영 가능 여부와는 무관한 기술적 참고치일 뿐, 실제 채택 여부는 2단계 사후 점검이 결정)

**2단계 — 머신러닝 기반 표본충분성 검증 + 사후 통합**
- K별로 XGBoost(또는 동일 계열 분류기)를 학습시켜 교차검증 성능 비교
- **[필수, 빠뜨리지 말 것] 사후 표본 점검**: 1단계가 제안한 각 K 구조에서 구간별 실제 표본수·이탈건수를 직접 집계 → **이탈 392건 미달 구간은 인접 구간과 통합 후 재검증** (4번에서 확정한 사전 범용 기준과 동일하게 적용 — 500건이 아닌 392건이 코드 전체에서 일관되게 쓰이는 임계값)
- 최종 K와 경계는 위 사후 점검을 통과한 구조로 확정
- **실데이터 사전 예측**: 6구간 균등분할 시뮬레이션 결과 후반 구간(43-58개월 168건, 58-72개월 118건)이 이미 392건 기준에도 미달 확인됨. 전체 표본규모(7,043건, 1만건 기준 미달)와 구간별 이탈률 불균등(54%→7%)이 겹쳐 **최종 K는 2~3개 정도로 수렴할 가능성이 높음** (사전 예측, PELT 실행 후 재확인 필요). 이는 설계 결함이 아니라 데이터 부족 시 스스로 보수적 결과로 수렴하는 의도된 안전장치로 해석

**세그먼트 이름**: 사전에 "신규/중간/장기"로 고정하지 않음. 경계 확정 후 분석 B로 각 구간 성격을 파악한 다음 사후에 이름 부여 (임시로는 "구간1, 구간2..." 사용)

## 4. 표본충분성 기준 (2단 구조)

**① 사전 범용 기준 (데이터셋 무관, 재사용 가능)**
- Cohen(1988) 표준 효과크기(작은 효과크기 h=0.2) + 유의수준 5% + 검정력 80%로 검정력 분석
- 계산 결과: 그룹당 약 **392건** → **코드에서 실제 사용할 임계값은 이 392건(검정력 분석 함수의 출력값)으로 확정**
  - ⚠️ [정정] 이전 버전에서 "보수적으로 500건"이라고 했던 건 논리적 일관성 위반이었음을 확인. 500건은 우리가 직접 "근거 없는 경험칙"으로 폐기 판정한 숫자라, 이를 다시 "더 안전해 보인다"는 직감으로 가져오면 2.7-C에서 비판한 패턴("사람이 안전해 보이는 숫자를 직감으로 정한다")을 그대로 반복하는 셈
  - 392라는 숫자를 하드코딩하지 말고, `NormalIndPower().solve_power(...)` 함수를 코드에 남겨 그 출력값을 임계값으로 사용 (가정값이 바뀌면 임계값도 자동으로 같이 바뀌도록)
- `statsmodels.stats.power.NormalIndPower` 사용 (사전 기준 계산용)

**② 사후 데이터 특화 검증 (현재 데이터에 맞춰 직접 확인)**
- 이탈 "건수"의 부트스트랩 변동계수(CV = 표준편차/평균)를 K별로 계산해 표로 제시
  - ⚠️ 평균 이탈률의 신뢰구간 폭은 쓰지 않음 (K=1이 항상 유리해지는 왜곡 확인됨, 폐기됨)
- 2단계 교차검증 성능도 K별로 함께 제시
- **임계값은 자동 탐지하지 않음** (2차 미분 기반 꺾임 탐지는 부트스트랩 시드 바꾸면 결론이 바뀌는 재현성 문제로 폐기 확인됨) → CV·성능 표를 투명하게 제시하고 사람이 종합 판단
- 500건은 코드의 임계값으로 쓰지 않되, 사람이 이 표를 검토할 때 "기존 업계 경험칙과 비교해 얼마나 차이나는지" 참고하는 보조 주석 정도로만 남겨둠 (의사결정 변수가 아님)

**⚠️ 7번(부트스트랩 안정성 검증)과의 관계 — 같은 기법, 다른 목적, 별도 실행**
- 이 섹션의 부트스트랩(CV 계산용)과 7번의 부트스트랩(경계 재현성 확인용)은 **같은 재추출 기법을 쓰지만 목적과 측정 대상이 다른 별도 절차**
  - 이 섹션: "K개 세그먼트로 나눴을 때 각 구간의 이탈 **건수**가 재추출해도 안정적인가" → 표본충분성 판단용
  - 7번: "PELT가 찾은 변화점의 **위치(경계 좌표)**가 재추출해도 같은 곳에서 나오는가" → 경계 자체의 안정성 판단용
- 코드 구현 시 두 부트스트랩 함수를 분리해서 작성 (재사용 가능하지만 자동으로 같은 결과를 공유하지 않음, 각각 따로 실행)

## 5. 예측모델 설계

**입력 피처 (확정)**
- 전체 속성 + **tenure 원본** + **세그먼트 라벨** (둘 다 사용, 양자택일 아님)
  - tenure 원본: 세밀한 정보 보존
  - 세그먼트 라벨: 분석 A 발견의 기여도를 SHAP으로 직접 검증 가능
  - 다중공선성 위험 인지하나 트리 기반 모델은 강건, 1단계 비교군(선형모델)만 영향 받음(치명적이지 않음)

**3단계 비교 구조 (입력 차이를 "세그먼트 라벨 포함 여부" 하나로 통제)**

| 단계 | 알고리즘 | 입력 피처 | 역할 |
|---|---|---|---|
| 1단계 | 로지스틱 회귀 | 전체 속성 + tenure 원본 (스케일링됨) | 전통통계 비교군 |
| 2단계 | XGBoost | 전체 속성 + tenure 원본 (세그먼트 라벨 미포함) | 현재AI 비교군 |
| 3단계 | XGBoost (2단계와 동일 알고리즘) | 전체 속성 + tenure 원본 + **세그먼트 라벨** | **메인, 우리의 접근** |
| 보조 | 간단한 MLP | 2단계와 동일 입력 (스케일링됨) | 딥러닝 비교용 (부가) |

**왜 1단계에도 세그먼트 라벨이 없는가 (신규 결정, 메모에 없던 빈틈 보충)**: 1단계는 2.10에서 "사람이 사전에 정한 단순 함수형태에 의존"하는 전통통계를 대표하는 비교군으로 정의됨. 세그먼트 라벨 자체가 "데이터 주도(분석 A)로 찾아낸 경계"이므로, 이를 1단계에 넣으면 "전통적/단순"이라는 1단계의 정의와 모순됨. 따라서 1단계는 세그먼트 라벨을 절대 포함하지 않음 (2단계와도 동일 조건이라 2·3단계 비교의 순수성도 유지됨)

**왜 2·3단계가 같은 알고리즘인가**: 성능 차이가 "접근법" 때문인지 "알고리즘" 때문인지 혼동되지 않게 하기 위함

**보조 분석 Q (클러스터링, 메인 모델과 무관)**
- tenure+속성을 함께 K-means/GMM으로 클러스터링 (전체 데이터, 세그먼트 구분과 무관하게 1회 수행)
- 목적: "어떤 속성 조합이 결합되면 위험한가"라는 인사이트만 추출, 새 세그먼트로 채택 안 함
- 모델 입력에는 쓰지 않음, 발표/해석 보강용

## 6. 평가지표 및 임계값

**보고서에는 전부 계산해서 표로 제시**: Accuracy, Precision, Recall, F1, F2, ROC-AUC, PR-AUC
- Accuracy는 참고용으로만 표시 (전부 비이탈로 찍어도 73.5% 나오는 불균형 데이터)

**모델 튜닝 목적함수**: F2-score(Recall에 2배 가중치) 우선, Recall 보조 모니터링
- 이유: FN(이탈자 놓침)이 FP보다 비싼 비용이라는 가정 (정확한 비용비율은 미해결, 가정값임을 명시)
- **베타값(2)과 비용비율의 관계 (신규 결정, 메모에 없던 빈틈 보충)**: F-beta의 베타는 "비용비율을 정밀하게 환산한 수치"가 아니라 근사적 가중치임. 베타=2는 Recall을 Precision보다 대략 4배(베타²) 중요하게 취급한다는 의미. 비용비율 5:1 같은 구체적 숫자와 베타=2가 수학적으로 정확히 대응하지는 않음 — 정밀한 비용 최적화가 필요하면 베타값 대신 비용 행렬(cost matrix)을 직접 모델 학습에 반영하는 방법이 더 정확하나, 본 프로젝트는 F2를 근사적 출발점으로 채택하고 정확한 비용비율은 민감도 분석으로 보강

**분류 임계값**: 0.5 고정 대신, Precision-Recall 곡선에서 "Recall이 급격히 꺾이기 직전 지점" 채택

---

## 7. 3단계 비교 — 실증 책임 (반드시 검증할 것)

1. **부트스트랩 안정성 검증**: 변화점 탐지 결과를 재추출 반복(1,000회 권장)해 경계가 안정적으로 재현되는지 확인
2. **1단계와 비교**: Kaplan-Meier(`lifelines` 라이브러리)와 PELT 변화점 탐지 결과 비교
3. **2단계와 비교 (가장 중요)**: 동일 데이터에 2단계(세그먼트 라벨 없음)와 3단계(세그먼트 라벨 포함)를 모두 적용해 "3단계가 2단계보다 추가로 무엇을 보여주는지" 입증
   - 성능 차이 발생 시: 세그먼트 발견이 실제로 기여했다는 직접 증거
   - 성능 차이 없을 시: SHAP으로 세그먼트 라벨 피처의 중요도를 보여주며 해석적 가치 강조 (성능이 아니라 "구조를 보여준다"는 서사로 전환)
4. **해석 보강**: 분석 B 결과로 "왜 이 경계 지점인가" 설명

---

## 8. 분석 B 및 보조 Q 구현 메모

- **분석 B**: 세그먼트 확정 후, 세그먼트별로 속성-이탈률 관계 탐색 (의사결정나무 분기, 교차표, 속성별 이탈률 비교)
- **순서 중요**: 분석 A → 세그먼트 확정 → 분석 B (세그먼트별) → 보조 Q. Q는 분석 A/B와 완전히 무관하게 전체 데이터로 1회만 수행
